## Goal

Build **MCProbe**, a stdio MCP server that audits other MCP servers over the live protocol: it connects to any MCP target (stdio or HTTP), lints every tool's schema for agent-usability, behaviorally fuzzes each tool with deliberately broken inputs, and returns a 0–100 conformance score with a per-dimension breakdown rendered as Markdown.

## Context

- The complete project specification lives at `docs/requirements/257dbd92dde4-mcprobe-spec.md` and is the single source of truth for scope, API, lint rules, scoring, demo, and acceptance criteria.
- The workspace is empty except for `docs/requirements/` and `.gitignore` (`node_modules/`, `dist/`, `.humanize/` are already ignored). No prior code, no `package.json`.
- MCProbe plays two MCP roles simultaneously:
  - **Server** to the host (Claude / IDE / agent) — exposes `probe_*` tools over stdio.
  - **Client** to the target under audit — speaks MCP to a third-party server.
- The four core tools (`probe_connect`, `probe_lint`, `probe_fuzz`, `probe_report`) map one-to-one to four pipeline stages (connect, lint, fuzz, score+render). Two optional helpers (`probe_list`, `probe_disconnect`) are explicitly "build only if time allows."
- Tech stack is frozen: TypeScript ESM, Node 18+, `@modelcontextprotocol/sdk`, `ajv` + `ajv-formats`, `zod`, `vitest`, `tsc` only (no bundler). Four runtime deps + one dev dep — strictly enforced.
- Strict vocabulary: use the project terminology in code, docs, and tool output. Always: MCP, tool, schema, JSON Schema, conformance, fuzz, lint, transport, stdio, capability, severity, finding.
- Behavioral fuzzing is the hero feature; static linting is the supporting act. The first sentence of every user-facing artifact must lead with behavior, not metadata.

## Assumptions

- Node 18+ and `npm` are available on the executor's PATH. The build target is CommonJS-free ESM (`"type": "module"`).
- The executor can reach `registry.npmjs.org` to install `@modelcontextprotocol/sdk`, `ajv`, `ajv-formats`, `zod`, and the dev-only `vitest` + `typescript`.
- The executor can `npx -y @modelcontextprotocol/server-filesystem /tmp` (or an equivalent third-party MCP server) for the AC-7 generality proof. If network policy blocks fetching the npm package at runtime, document the constraint and use a local mock target (clearly labeled as a degraded alternative) — the **primary path is the real external server**, captured in `examples/transcripts/external-server.md`.
- The MCP SDK's `Client` + `StdioClientTransport` / `StreamableHTTPClientTransport` (or current `SSEClientTransport` if streamable HTTP is not yet stable on the resolved SDK version) APIs are used as published. If the resolved SDK version renames or splits transports, pick the documented stable one and leave a short note in `claude.md` (not in tool output).
- "Protocol crash" in fuzzing means the underlying `Client.callTool` promise rejects with a transport/JSON-RPC error, or the transport emits a close/disconnect event before a tool result arrives. The probe treats this as distinct from a well-formed `isError: true` tool result.
- Scoring: the two static dimensions are subtractive (start at 10, lose points per finding); the two behavioral dimensions are normalized rates (graceful-rejection rate for Error Handling, valid-success rate for Liveness). Overall score is the mean of *measured* dimensions; unmeasured (e.g. behavioral ones when `fuzz: false`, or when every tool was skipped) are reported as "not measured" and excluded from the average.
- Demo target is a sibling package under `examples/demo-target/` with its own `package.json` and `tsconfig.json`, built and run with `node dist/index.js` (or `tsx` for dev). It re-uses the project's ESM/TS settings.
- All tool handlers wrap their body in `try/catch` and return MCP error results with `isError: true` on failure; they never throw out.
- Logging is to **stderr only** — stdout carries the JSON-RPC stream of the probe's own stdio server.

## Implementation Notes

### Approach selection (briefly)

Three viable shapes were considered:
1. **Monolithic `index.ts` with inline logic** — fastest to write, impossible to test in isolation, violates spec §5's pure-logic separation.
2. **Full DI container with injectable transports** — clean but overkill for a four-tool probe and adds dependencies the spec forbids.
3. **Layered modules with a single shared `ConnectionRegistry` (chosen)** — matches spec §5's diagram exactly, makes the lint/fuzz/score/report modules pure and trivially unit-testable, and keeps the I/O surface in exactly two files (`index.ts` and `target-client.ts`). Selected.

### Work batches (sequenced; each batch ends with a runnable checkpoint)

**Batch 1 — Skeleton & build** (AC-1, AC-12)
- Author `package.json` with exactly the four runtime deps and `vitest` + `typescript` + `@types/node` as devDeps. `"type": "module"`, `"bin": { "mcprobe": "dist/index.js" }`.
- Author `tsconfig.json`: `target: ES2022`, `module: NodeNext`, `moduleResolution: NodeNext`, `strict: true`, `outDir: dist`, `rootDir: src`, `declaration: false`.
- Add `scripts`: `build: tsc`, `start: node dist/index.js`, `dev: tsc --watch`, `test: vitest run`.
- Stub `src/types.ts` and an empty `src/index.ts` that registers zero tools. Build must succeed.

**Batch 2 — Target client** (AC-3)
- `src/target-client.ts` exports:
  - `connectStdio({ command, args?, env? })` → `Connection` (wraps SDK `Client` + `StdioClientTransport`).
  - `connectHttp({ url })` → `Connection` (wraps `StreamableHTTPClientTransport` or current SSE transport, whichever is stable in the resolved SDK).
  - `ConnectionRegistry`: in-memory `Map<connectionId, Connection>` plus a `defaultId` set to the most recent successful connect. `getDefault()` returns it or throws a clean error.
  - `listTools(connection)`, `callTool(connection, name, args)` returning `{ ok, isError, content, error?, latencyMs }`.
  - `disconnect(id?)` and `disconnectAll()`.
- Inherit parent env for stdio children, then layer caller `env` on top (spec §8).
- All log lines go to `console.error` (stderr).

**Batch 3 — Probe server & `probe_connect`** (AC-2, AC-3)
- `src/index.ts` uses `McpServer` from `@modelcontextprotocol/sdk/server/mcp.js`, registers tools via `server.tool(name, description, zodSchema, handler)`.
- Register `probe_connect` with zod input schema mirroring spec §4.1; handler calls `target-client`, stores in registry, returns the structured handshake summary.
- All handlers wrap their body in `try/catch` and return `{ isError: true, content: [{ type: "text", text: <readable message> }] }` on failure (spec §8).

**Batch 4 — `probe_lint`** (AC-4)
- `src/schema-lint.ts`: pure function `lintTools(tools: ToolSummary[]): Finding[]` implementing all 12 rules from spec §6 with stable `code`, `severity`, `message`, `location` (`{ tool, param? }`), and `hint`.
- Ajv is instantiated with `strict: false`, `allErrors: true`; compile failures produce `schema.invalid`.
- Register `probe_lint` in `index.ts`; default to the most recent connection.

**Batch 5 — `probe_fuzz`** (AC-5)
- `src/fuzz.ts`: pure `generateCases(schema: JSONSchema): FuzzCase[]` returning one `valid` and at least three malformed variants per tool:
  - `missing_required` — drop each required field one at a time.
  - `wrong_type` — for each typed property, send a value of a different primitive type.
  - `out_of_enum` — for each `enum` property, send a value not in the enum.
  - `extra_garbage` — append a clearly non-schema key (e.g. `__mcprobe_garbage__: true`).
  - For tools with no schema, send `{}` and `{ "__mcprobe_garbage__": "x" }` and treat both as malformed.
- `runFuzz(connection, tools, options)`: skip tools annotated `destructiveHint: true` unless `options.fuzzDestructive`, cap the rest at `options.maxTools` (default 10), run every case per fuzzed tool, and return `{ results, coverage }`. Each result is `{ name, case, outcome: "ok"|"toolError"|"protocolCrash", silentlyAccepted, latencyMs, errorMessage? }`; `coverage` is `{ totalTools, fuzzedTools, skippedDestructive, skippedOverCap }`.
  - `silentlyAccepted` is `true` when the case was malformed *and* the target returned `isError: false` with any content (i.e. the tool did not complain).
  - `protocolCrash` is set when the `callTool` promise rejects or the transport closes mid-call; the function then attempts a re-list and re-initialize on the same connection before continuing to the next tool.

**Batch 6 — `probe_report` & scoring** (AC-6)
- `src/conformance.ts`:
  - `scoreMetadata(serverInfo, capabilities)` — full marks (10) for name+version, +1 bonus point (clamped to 10) for `instructions`.
  - `scoreSchemaQuality(findings)` — start at 10, subtract 1 per `error`, 0.5 per `warning`, 0.25 per `info`. Clamp to [0, 10].
  - `scoreErrorHandling(fuzzResults)` — normalized rate over **malformed** cases: `10 × (gracefully-rejected / total malformed)`. Silent accepts and protocol crashes both count as failed rejections. Clamp.
  - `scoreLiveness(fuzzResults)` — normalized rate over **valid** cases: `10 × (successful / total valid)`, minus 0.5 per 100ms the valid-call p50 latency exceeds a 200ms target. Clamp. (Malformed cases score Error Handling and valid cases score Liveness — the case kinds are partitioned so no outcome is double-counted.)
  - `rollup(scores)` — return `{ measured: scores with `notMeasured: false`, overall: round(mean * 10) }` and a letter grade via the A/B/C/D/F thresholds in spec §7.
- `src/report.ts`: pure `renderReport({ server, dimensions, findings, fuzz })` returns Markdown with the exact sections required by AC-6.
- Register `probe_report` in `index.ts`; it runs introspect + lint and (if `fuzz: true`) fuzz, then renders.

**Batch 7 — Optional helpers** (only if Batch 6 lands with time to spare)
- `probe_list` — `tools/list` on the default connection, returns `name`, `description`, `inputSchema` per tool.
- `probe_disconnect` — close one or all connections. Omit `id` to close all.

**Batch 8 — Demo target** (AC-3, AC-4, AC-5)

  - `well_behaved` — full description, schema validates, returns `isError: true` on bad input.

### Key technical decisions and risks

- **SDK transport selection for HTTP targets** — risk: the SDK's streamable-HTTP transport may not be stable on the resolved version. Mitigation: prefer `StreamableHTTPClientTransport` if exported; fall back to `SSEClientTransport`; document the choice in `claude.md`. The build does not block on this; the HTTP path is exercised by AC-3 only if an HTTP target is used, and the default demo is stdio.
- **Reconnect-after-protocol-crash** — required by spec §5's "guards against method-not-found" principle. A `protocolCrash` on tool *N* must not abort the audit of tools *N+1..M*. Implementation: catch in `runFuzz`, call `connection.refresh()` (or `client.listTools()` to verify liveness), and continue.
- **Latency measurement** — use `process.hrtime.bigint()` deltas; report p50 and max per tool in the fuzz table. Do not include the generator's own time.
- **Project terminology** — enforce via a tiny `scripts/check-readme.mjs` and an inline check in `claude.md` linter. Treat any drift as a build-blocking error.
- **Demo target determinism** — `divide` and `greet` must produce the same output across runs (no randomness, no `Date.now()`) so the smoke assertions are stable.
- **No bundler** — `tsc` output is shipped directly; `package.json` `"main": "dist/index.js"` and `"bin": { "mcprobe": "dist/index.js" }` point at the compiled entry.
- **Stdout hygiene** — every `console.log` in `src/` is a bug. Use `console.error` (or a tiny `log.ts` wrapper that writes to stderr) for any operator-visible log. The probe's own `McpServer` writes its JSON-RPC to stdout; we never interleave.

### Browser / web tasks

This plan does not require any browser operations, web scraping, form filling, page navigation, or screenshots. The `agent-browser` skill is therefore not used in the executor's run. The "external server" target in AC-7 is an npm-distributed MCP server invoked via `npx` over stdio, not a web page.

## Out of Scope

- Remaining spec §13 roadmap items: lint/fuzz of resources and prompts, `probe_report --format json`, and a GitHub Action. (The `destructiveHint`-aware fuzz dry-run has since been implemented.)
- Persistence: no database, no on-disk connection state beyond the captured transcripts.
- A web UI or HTTP server for MCProbe itself. The probe is stdio-only; HTTP appears only as a *target* transport.
- Auth flows beyond the documented `env` passthrough for stdio children and a plain URL for HTTP targets.
- Deep schema generation: best-effort sampling for nested/`oneOf` inputs is sufficient. No schema-aware property-value synthesizers beyond the primitives in spec §4.3.
- Resource and prompt *auditing* (linting/fuzzing). They are still *counted* in the connect handshake; that is the entire scope.
- More than four runtime dependencies. AC-12 is a hard gate.
